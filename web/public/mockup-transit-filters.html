Transit & Distance Filter Options

Four design approaches for adding subway proximity and commute-time filters to the Dwelligence filter bar. All options use pre-computed isochrone data so filtering is instant (no API calls on slider change). Pick A, B, C, or D below.

Pros
  • Most powerful: users build their own filter tabs from real destinations
  • Removes hardcoded geography — scales to any city

D Hybrid
Keep the existing area tabs as built-in presets, but add a "Near Subway" chip in the filter row. Clicking an area tab auto-populates a commute filter behind the scenes. Users can also add custom destinations.

When user clicks "Fulton St" area tab, the app internally applies the equivalent isochrone filter (25 min transit to Fulton St station). The user can then open Near Subway to fine-tune walk distance. If they click a custom destination tab like "100 Broadway", that filters by its commute isochrone instead. Area tabs and custom destinations are mutually exclusive selections (radio behavior).
Pros
  • Zero breaking changes — existing area tabs work exactly as before
  • Gradual migration: users discover custom destinations over time
  • Subway filter is independently useful even without commute setup
  • Area tabs gain superpowers (backed by isochrone data) without UI change
  • Moderate implementation effort — builds on existing architecture
Cons
  • Two concepts in one dropdown (subway + destinations) may feel crowded
  • Custom destination tabs + area tabs in one row could get long
  • Radio behavior between preset/custom tabs needs clear visual distinction
