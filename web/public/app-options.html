TL;DR

Start with a PWA (Option A) now — it takes 2-3 days, costs nothing, and gets you home screen install + offline support immediately. Then build with Expo (Option D) when you need App Store presence and real push notifications. Expo gives you native gestures (critical for the swipe feature), file-based routing you already know from Next.js, and stays entirely in JS/TS. Skip React Native bare workflow, Capacitor, and native Swift/Kotlin — they either add complexity you don't need or require languages you don't use.

What it takes

  • manifest.json — Add a web app manifest with name, icons (192px, 512px), theme color, display: standalone. Next.js 15 supports this natively via app/manifest.ts.
  • Service Worker — Use next-pwa or @serwist/next to auto-generate a service worker. Handles caching, offline fallback pages, and asset precaching.
  • Offline support — Cache the app shell + recently viewed listings. API calls can use a cache-then-network strategy. Supabase offline mode is limited, so focus on read-only offline viewing.
  • Add to Home Screen — Works on Android (Chrome shows install banner automatically). On iOS Safari, users tap Share > Add to Home Screen. No browser chrome in standalone mode.
  • Push Notifications — Web Push API works on Android Chrome and desktop. iOS Safari added web push in iOS 16.4 (2023), but only for PWAs added to home screen. Requires user to re-open the PWA, not just visit the URL.
  • Icons & Splash — Need Apple touch icons and meta tags for iOS splash screens. Can use pwa-asset-generator to auto-create these from a single source image.

Vercel compatibility

PWAs work fine on Vercel. The service worker is served as a static file. No infrastructure changes needed.

Pros

  • Zero new codebase — enhance what you have
  • Done in a weekend
  • No app store review process
  • Instant updates (no app store delay)
  • Same URL, SEO preserved
  • Your @use-gesture swipe code works as-is
  • Leaflet maps work as-is
  • Framer Motion animations preserved

Cons

  • No App Store / Play Store listing (discoverability)
  • iOS push notifications are janky — require home screen install, can break on iOS updates
  • iOS Safari quirks: no background fetch, limited storage (50MB), can be evicted after 7 days of no use
  • Swipe gestures are good but not native-feel (60fps vs 120fps, no haptic feedback)
  • No access to native APIs (haptics, contacts, NFC, etc.)
  • Users unfamiliar with PWA install flow
  • Google Play Store listing possible via TWA (Trusted Web Activity) but adds complexity

What you'd build

  • Supabase backend reuse — @supabase/supabase-js works identically in React Native. Auth, database queries, realtime subscriptions — all portable. This is the biggest win.
  • UI rebuild — React Native uses <View>, <Text>, <ScrollView> instead of HTML divs. No Tailwind (you'd use nativewind or StyleSheet.create). Your component logic transfers but JSX does not.
  • Maps — Replace Leaflet with react-native-maps (uses Apple Maps on iOS, Google Maps on Android). Better performance than web maps. Supports markers, clusters, custom overlays.
  • Gestures — react-native-gesture-handler + react-native-reanimated give you 120fps gesture animations running on the native UI thread. This is the gold standard for swipe interactions.
  • Navigation — @react-navigation/native with stack + tab navigators. Different paradigm from Next.js file routing.
  • Push notifications — react-native-firebase or @notifee/react-native for full native push via APNs (iOS) and FCM (Android).

Key libraries

  • react-native-maps — Native map views
  • react-native-gesture-handler — 120fps gestures on native thread
  • react-native-reanimated — Performant animations
  • nativewind — Tailwind for React Native (partial compatibility)
  • @supabase/supabase-js — Same client you already use
  • react-native-fast-image — Cached image loading

Cost breakdown

Apple Developer Account: $99/year. Google Play Console: $25 one-time. No other infrastructure costs — you're still using Supabase + Vercel for the API/backend.

Pros

  • Best swipe gesture quality (native thread animations)
  • Full App Store + Play Store presence
  • Native push notifications (reliable, rich)
  • Access to all native APIs
  • Same language (JS/TS) you already know
  • Supabase client works unchanged
  • Large ecosystem and community

Cons

  • UI must be completely rebuilt (no HTML/CSS)
  • Two codebases to maintain (web + native)
  • Bare workflow requires Xcode + Android Studio setup
  • Native module linking can break on RN upgrades
  • Slower iteration than web (build + deploy cycle)
  • Need to manage iOS/Android signing, certificates
  • No Framer Motion — need to learn Reanimated API

How it works

  • Concept — Capacitor creates iOS/Android projects that embed a WKWebView (iOS) or Android WebView. Your Next.js app runs inside this WebView, but with a JavaScript bridge to native APIs.
  • Setup — Run npx cap init, export your Next.js app as static (output: 'export' in next.config.js), then npx cap add ios and npx cap add android. The exported HTML/JS/CSS gets bundled into the native app.
  • Important limitation — Next.js server-side features don't work in Capacitor. No API routes, no SSR, no server components. You'd need to either: (a) use output: 'export' for a fully static build, or (b) point the WebView at your Vercel URL (live mode). Option (b) is easier but requires internet.
  • Native plugins — @capacitor/push-notifications for native push, @capacitor/haptics for vibration feedback on swipes, @capacitor/camera, @capacitor/geolocation, etc.
  • Maps — Leaflet continues to work inside the WebView. You could optionally use @capacitor/google-maps for a native map layer rendered behind the WebView (better perf, but more complex).
  • Gestures — Your existing @use-gesture + Framer Motion swipe code works. Add @capacitor/haptics for tactile feedback. Gesture performance is WebView-bound (good, not native-level).

The SSR problem

Your current app likely uses Next.js server components and API routes. Capacitor needs either a static export or a live URL. The pragmatic approach: point Capacitor's WebView at your production Vercel URL. The app works exactly like the web version, but wrapped in a native shell with push notification access. The downside is no offline support.

Pros

  • 95% code reuse — your web app IS the mobile app
  • Native push notifications via Capacitor plugin
  • App Store distribution
  • Haptic feedback for swipes (native plugin)
  • Fast to set up (days, not weeks)
  • Leaflet maps work unchanged
  • Framer Motion animations preserved
  • Well-maintained by Ionic team

Cons

  • WebView performance ceiling (not true native)
  • Apple has rejected low-effort WebView wrappers — need to add native value
  • Next.js SSR/API routes require live URL or architectural changes
  • Swipe gestures are good, not excellent (WebView thread)
  • Debugging WebView issues is frustrating
  • Two build pipelines (web + native)
  • Need Xcode + Android Studio for builds

Why Expo is the sweet spot

  • Expo Router — File-based routing that works like Next.js App Router. app/(tabs)/search.tsx, app/listing/[id].tsx — you already think this way. Supports layouts, nested routes, typed routes.
  • EAS Build — Cloud build service. No need to own a Mac for iOS builds (though having one helps for testing). Builds are submitted directly to App Store Connect and Google Play Console. Free tier: 30 builds/month.
  • EAS Update (OTA) — Push JS bundle updates instantly without going through app store review. Like Vercel deployments but for mobile. Fix bugs in minutes, not days.
  • Expo Notifications — expo-notifications handles APNs + FCM setup, token management, and rich notification content. Push service is free for reasonable volume.
  • Gestures — Expo includes react-native-gesture-handler and react-native-reanimated by default. Your swipe card interaction would run on the native UI thread at 120fps. This is categorically better than web gestures.
  • Maps — react-native-maps is in the Expo SDK. Apple Maps on iOS (free), Google Maps on Android (free up to very high usage). Native map performance far exceeds Leaflet.
  • Dev experience — npx expo start, scan QR code with Expo Go, see changes instantly on your phone. Hot reload. No Xcode/Android Studio needed during development (only for custom native modules).
  • Auth — @supabase/supabase-js works in Expo. Use expo-secure-store for token persistence (replaces browser localStorage).

What transfers from your current codebase

  • 100% reusable: Supabase queries, API logic, type definitions, business logic, state management (React Query)
  • Needs adaptation: Tailwind classes → NativeWind (similar syntax), Framer Motion → Reanimated (different API, same concepts), @use-gesture → gesture-handler
  • Needs rebuild: JSX (div/span/p → View/Text/ScrollView), Leaflet components → react-native-maps, Next.js-specific patterns (server components, API routes)

Cost breakdown

Apple Developer: $99/year. Google Play: $25 one-time. EAS Build free tier is sufficient for a solo developer. EAS Update free tier: 1,000 updates/month.

Pros

  • File-based routing — familiar mental model from Next.js
  • 120fps native gestures (best possible swipe experience)
  • EAS Build — no local Xcode/Android Studio needed for builds
  • OTA updates — deploy fixes without app store review
  • Native push notifications (reliable, rich)
  • Native maps (Apple Maps / Google Maps)
  • All JavaScript/TypeScript — no Swift/Kotlin needed
  • Huge ecosystem, excellent docs, active development
  • Expo Go for instant development testing
  • Supabase integration is first-class

Cons

  • UI needs to be rebuilt (no HTML/CSS)
  • 6-8 weeks of work to reach parity
  • Two codebases to maintain long-term
  • Learning curve for Reanimated (replaces Framer Motion)
  • Some Expo SDK limitations if you need very custom native modules (can eject to bare)
  • React Native upgrade cycle can introduce breaking changes

What this means

  • Two completely separate apps — iOS in Swift/SwiftUI, Android in Kotlin/Jetpack Compose. Both connect to the same Supabase backend but share zero UI code.
  • Learning curve — As a JS/TS developer, you'd need to learn Swift (memory management, optionals, protocols, SwiftUI declarative patterns) and Kotlin (coroutines, null safety, Compose). This is months of learning, not weeks.
  • Maps — MapKit (iOS) and Google Maps SDK (Android). Best possible map performance and platform integration.
  • Gestures — UIKit gesture recognizers (iOS) and MotionLayout/GestureDetector (Android). Platform-perfect swipe behavior with full haptic API access.
  • Supabase — Official Swift and Kotlin clients exist (supabase-swift, supabase-kt). They work, but the API surface differs from the JS client you know.

When this makes sense

Native makes sense when: (a) you have a funded team with dedicated iOS and Android developers, (b) your app needs deep OS integration (widgets, Siri shortcuts, Android auto, etc.), or (c) performance is so critical that even React Native's bridge overhead matters (think real-time gaming or video editing). For a property search app, this is overkill.

Pros

  • Best possible performance on each platform
  • Perfect platform-native feel and gestures
  • Full access to every OS API, day-one
  • Apple/Google love native apps in store reviews
  • Smallest binary sizes
  • No abstraction layer overhead

Cons

  • Two new languages to learn (Swift + Kotlin)
  • Three codebases to maintain (web + iOS + Android)
  • 16-24 weeks of development minimum
  • Features must be implemented twice
  • Hiring native developers is expensive
  • Almost no code shared with web app
  • Massive overkill for current app complexity

How it works

  • Tauri 2.0 — Released stable in late 2024. Uses the system WebView (WKWebView on iOS, Android WebView) to render your web app, with a Rust backend for native API access. Think Capacitor, but the native bridge is Rust instead of Java/Swift.
  • Bundle size — Significantly smaller than Capacitor or Electron because it uses the system WebView instead of bundling Chromium. Typical app: 5-10MB vs 50-100MB.
  • Rust knowledge — You need some Rust for custom native functionality. The plugin system uses Rust. As a JS developer, this means learning ownership, lifetimes, cargo (think npm but for Rust), and the borrow checker. The learning curve is steep but the Tauri API abstracts most of it.
  • Mobile support maturity — Tauri mobile (iOS + Android) is production-ready as of Tauri 2.0, but the ecosystem is much smaller than Capacitor's. Fewer plugins, fewer Stack Overflow answers, smaller community for mobile specifically.
  • Same SSR problem as Capacitor — Next.js server features don't work in a WebView. Same solution: static export or live URL.

Plugin ecosystem

  • tauri-plugin-notification — Local and push notifications (less mature than Capacitor's plugin)
  • tauri-plugin-geolocation — GPS access
  • tauri-plugin-haptics — Vibration feedback
  • tauri-plugin-http — Native HTTP client (bypass CORS)
  • Smaller plugin library overall — may need to write Rust plugins for missing features

Pros

  • Tiny bundle size (5-10MB)
  • 90% code reuse from web app
  • Better security model than Capacitor (Rust backend)
  • Multi-platform: iOS, Android, macOS, Windows, Linux from one project
  • Rust backend is fast for heavy computation
  • App Store distribution
  • Growing community and corporate backing

Cons

  • Rust learning curve (steep for JS developers)
  • Mobile ecosystem is young — fewer plugins than Capacitor
  • Push notification plugin is less mature
  • Same WebView gesture limitations as Capacitor
  • Same SSR/server component problems as Capacitor
  • Debugging can be harder (Rust + WebView + native layers)
  • Smaller community = fewer answers when you're stuck
  • Apple app review risk (WebView wrapper)

Maps: Leaflet vs Native Equivalents

Leaflet is web-only. Here's what replaces it in each approach, and what changes.

Approach Map Library Underlying Engine Migration Effort Notes
PWA Leaflet (unchanged) OpenStreetMap tiles Zero Works as-is in browser
Capacitor Leaflet (in WebView) OpenStreetMap tiles Zero Optional: @capacitor/google-maps for native layer
Tauri Leaflet (in WebView) OpenStreetMap tiles Zero Same as Capacitor
Expo / RN react-native-maps Apple Maps (iOS) / Google Maps (Android) Medium Different component API. MapView, Marker, Callout instead of MapContainer, TileLayer, Marker. Clustering via react-native-map-clustering.
Native iOS MapKit Apple Maps High SwiftUI Map view. Complete rewrite.
Native Android Google Maps SDK Google Maps High Jetpack Compose GoogleMap. Complete rewrite.

Map feature parity checklist

  • Markers with popups — Supported in all options. react-native-maps uses Callout instead of Popup.
  • Marker clustering — Leaflet uses leaflet.markercluster. React Native uses react-native-map-clustering. Both work well.
  • Custom tile layers — Leaflet is tile-agnostic. react-native-maps defaults to platform maps but supports custom tile overlays.
  • Gesture interaction — All options support pan/zoom. Native maps feel smoother due to native rendering.
  • Performance with many markers — Native maps handle 1000+ markers better than Leaflet. Leaflet can lag at 500+ without clustering.

What I'd Recommend

Two-phase approach: PWA now, Expo later

Given your constraints (solo JS/TS developer, swipe-centric app, need push notifications, Supabase backend), here's the practical path forward.

Phase 1 — This week

Ship a PWA

  • Add manifest.ts to your Next.js app (30 minutes)
  • Add @serwist/next for service worker generation (2-3 hours)
  • Create offline fallback page and cache strategy (2-3 hours)
  • Generate PWA icons from your logo (30 minutes)
  • Add iOS meta tags for splash screens and status bar (1 hour)
  • Test on Android Chrome (install prompt) and iOS Safari (Add to Home Screen)
  • Total: 2-3 days. Cost: $0. Result: installable app on every phone, today.

This gets you 80% of the mobile app experience with 1% of the effort. Your swipe feature, maps, auth, everything works. Push notifications work on Android immediately and on iOS for home-screen-installed users.

Phase 2 — When you're ready to invest

Build with Expo

  • Create a new Expo project alongside the web app
  • Set up Expo Router with the same route structure as your Next.js app
  • Port Supabase client, React Query hooks, and business logic (copy-paste)
  • Rebuild UI components with React Native + NativeWind (Tailwind syntax)
  • Replace @use-gesture + Framer Motion with gesture-handler + Reanimated (the big learning curve, but the payoff is 120fps swipe cards)
  • Replace Leaflet with react-native-maps
  • Add expo-notifications for native push
  • Submit to App Store and Play Store via EAS Build
  • Total: 6-8 weeks. Cost: $124/year (dev accounts). Result: native app in both stores.

Expo is the right choice over bare React Native because: managed builds (no Xcode hell), OTA updates (deploy like web), file-based routing (you already think this way), and the Expo SDK includes everything you need out of the box. You stay 100% in TypeScript.

Why not Capacitor?

Capacitor is tempting (95% code reuse, 1-2 weeks), but there are two problems: (1) your swipe feature is central to the app, and WebView gestures will always feel slightly off compared to native — this matters for a Tinder-style card interface; (2) Apple has been increasingly rejecting thin WebView wrappers, and you'd need to add meaningful native functionality to pass review. If swipe quality wasn't critical, Capacitor would be the answer.

Why not Tauri?

Tauri Mobile is technically impressive (small bundles, Rust backend), but the mobile ecosystem is still young. The push notification plugin is less mature, the community is smaller, and you'd need to learn Rust for any custom native functionality. As a JS/TS developer, this adds friction that Expo doesn't. Revisit Tauri in 2027 when the mobile story is more mature.

Why not Native?

Learning Swift + Kotlin to build two separate apps for a property search tool is not a good use of your time. React Native (via Expo) gets you 95% of native quality at 30% of the effort, entirely in the language you already know. Go native when you have a team, not when you're solo.
